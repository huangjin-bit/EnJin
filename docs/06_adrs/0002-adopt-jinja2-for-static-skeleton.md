# ADR-0002: 采用 Jinja2 作为静态骨架模板引擎

- **状态:** 已采纳 (Accepted)
- **日期:** 2026-03-16
- **决策者:** EnJin 架构组

## 背景 (Context)

EnJin 编译器需要将 I-AST 转化为目标语言代码（Python/Java）。
代码生成分为两类：
1. **基础设施层 (Infrastructure):** 数据库连接、入口文件、全局异常处理等——必须 100% 确定性。
2. **业务逻辑层 (Business Logic):** process 意图对应的代码——由 AI 动态生成。

需要一种模板机制来实现"静态骨架 + 动态插槽"的三明治组装。

## 决策 (Decision)

**采用 Jinja2 模板引擎**。

理由：
1. **成熟的 Python 生态：** Jinja2 是 Python 世界最广泛使用的模板引擎，文档完善。
2. **插槽机制天然契合：** `{{ process_code }}`、`{{ endpoint_handlers }}` 等受控插槽可精确标记 AI 或编译器生成内容的注入点。
3. **模板继承与宏：** `{% extends %}` 和 `{% macro %}` 支持目标语言模板的复用与分层。
4. **自动转义可关闭：** 生成代码时关闭 HTML 转义，直接输出原始字符串。
5. **条件渲染：** `{% if ... %}` 可在模板内部处理细粒度差异（如数据库方言、中间件版本）。注：编译单元模型确立后，同一编译单元只有一个目标栈，跨语言条件渲染不再适用于一级目标选择。

排除方案：
- **字符串拼接：** 不可维护，且容易引入缩进错误。
- **Mako：** 功能强大但语法晦涩，社区活跃度不如 Jinja2。
- **代码生成 AST (如 Python ast 模块)：** 仅限 Python 目标，无法支持 Java。

## 后果 (Consequences)

- **正面：** 基建层代码（database.py, main.py）由人工审计的模板硬编码生成，杜绝 AI 幻觉。
- **正面：** AI 生成的代码被严格限定在 `{{ slot }}` 插槽中，实现物理隔离。
- **正面：** 新增目标语言只需添加一套 `.jinja` 模板目录，不改编译器核心逻辑。
- **负面：** Jinja2 模板的调试体验较差（模板渲染错误的堆栈不够直观）。
- **缓解措施：** 编写模板渲染的单元测试，确保每个模板都有对应的 golden file 比对测试。
