# 蓝绿数据库迁移机制 (Blue-Green Migration)

> 本文档描述 EnJin 处理数据库 Schema 变更的安全策略。

---

## 核心原则

**严禁生成破坏性 DDL。** 当编译器检测到 `struct` 定义发生变更（字段新增/删除/改类型），
不允许直接生成 `ALTER TABLE ... DROP COLUMN` 等不可逆操作。

## 迁移流程

```
1. 检测到 struct 变更 (AST diff)
2. 创建影子表 (shadow table)
3. 双写期：新旧表同时写入
4. 数据同步：后台迁移旧表数据到影子表
5. 灰度切流：逐步将读流量切到新表
6. 旧表归档：确认无流量后归档旧表
```

---

## 变更检测

编译器通过比较当前 `struct` AST 与 `enjin.lock` 中缓存的上一次 `struct` 定义来检测变更：

| 变更类型 | 风险级别 | 处理策略 |
|---|---|---|
| 新增字段（nullable 或有 default） | 低 | 允许直接 `ALTER TABLE ADD COLUMN` |
| 新增字段（not null 无 default） | 高 | 要求影子表迁移 |
| 删除字段 | 高 | 要求影子表迁移 |
| 字段类型变更 | 高 | 要求影子表迁移 |
| 仅新增索引 | 低 | 允许直接 `CREATE INDEX` |

## 目标栈差异

- **Java / Spring（电商核心）：** 迁移脚本建议生成 Flyway / Liquibase 格式
- **Python / FastAPI：** 迁移脚本建议生成 Alembic 格式

## 中间件影响

高风险迁移期间还需考虑：

- Redis 缓存：涉及变更字段的缓存键需失效或双版本兼容
- MQ 消费者：若消息 payload 含变更字段，需做 schema 版本兼容
- 搜索索引：若 ES/Meilisearch 索引含变更字段，需重建索引

## 与编译单元模型的关系

- 迁移脚本按编译单元独立生成
- 不同编译单元的 struct 变更不会交叉影响
- 共享契约中的 DTO 变更不直接触发数据库迁移
